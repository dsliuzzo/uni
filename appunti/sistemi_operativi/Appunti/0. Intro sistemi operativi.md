# Struttura dei sistemi di calcolo
>[!info] Sistema operativo
>1. Un sistema operativo è un programma che agisce da **Intermediario** tra gli utenti e l'hardware di un computer con l'obbiettivo di:
>   - Eseguire i programmi dell'utente e aiutare gli utenti a risolvere i loro problemi tramite computer
>   - rendere i computer convenienti da usare
>   - usare l'hardware dei componenti in modo efficiente
>2. Il sistema operativo è un **allocatore di risorse**:
>   - gestisce tutte le risorse hardware e software
>   - gestisce le richieste in conflitto tra loro in modo da garantire un uso efficiente ed equo delle risorse
>3. Un sistema operativo è un programma di controllo
>   - Controlla l'esecuzione dei programmi in modo da prevenire errori nell'uso del computer

>[!important] Kernel
>programma sempre in esecuzione nel computer

>[!important] Bootstrap
>caricato all'accensione prende il controllo del sistema per trovare e caricare il **kernel** dando a lui il controllo.
>E' salvato su una ROM o EPROM.

## Storia dei sistemi
### Sistema mainframe
Primo computer con un sistema operativo. Aveva l'obbiettivo di automatizzare il passaggio da un **job** ad un altro tramite il **monitor residente**.
>[!important] Monitor residente
>Prima forma di sistema operativo. Rimaneva sempre in esecuzione con il solo compito di permettere l'utilizzo di programmi in batch.

![[Pasted image 20260306165048.png|center|300]]

Si passa così da sistemi monoprogrammati a sistemi multiprogrammati.
Abbiamo così una suddivisione dei programmi in più job.
All'accensione il monitor residente prenderà il controllo del sistema, per passare il controllo al primo job, finito il primo passerà il controllo (anche in modo non sequenziale). La RAM veniva quindi caricata con più programmi contemporaneamente.

Il lavoro del sistema operativo diventa quindi:
- Routine di I/O
- Memory management
- CPU scheduling
- Allocazione dei devices ai job

Questo ha permesso la creazione di sistemi in **time-sharing** che gestivano l'accesso di multipli utenti dividendo il carico del processore tra loro.
### SO per diversi calcolatori
#### Desktop - PC
Computer dedicati ad un singolo utente.
Può eseguire diversi sistemi operativi.
#### Sistemi paralleli
Sistemi multiprocessore, che condividono la memoria offrendo prestazioni superiori, economicità e affidabilità.
- SMP - sistema multiprocessore simmetrico
  ogni processore esegue una copia identica del sistema operativo
  ![[Pasted image 20260308123413.png|center|300]]
- AMP - sistema multiprocessore asimmetrico (master-slave)
  ogni processore esegue un compito specifico
  ![[Pasted image 20260308123427.png|center|300]]
#### Sistemi distribuiti
Distribuiscono l'elaborazione tra diversi calcolatori.
Aumentano le prestazioni, ma necessitano di una infrastruttura di rete.
Possono usare un modello client/server o peer-to-peer.
#### Cluster di computer
Un cluster di computer è composto da un insieme di computer convenzionali connessi in rete per costituire un singolo sistema parallelo.
![[Pasted image 20260308123720.png|center|300]]
#### Sistemi real-time
Sistemi con vincoli temporali ben definiti sull'elaborazione e sull'accesso alle risorse tramite algoritmi di scheduling specifici.
### Moderno sistema di calcolo
![[Pasted image 20260308123855.png|center|500]]
- I dispositivi di I/O e la CPU possono eseguire operazioni in modo concorrente.
- Ogni device controller è responsabile di un particolare tipo di dispositivo.
- Ogni device controller ha un buffer di memoria locale.
- La CPU muove i dati dalla/alla memoria centrale ai/dai buffer locali.
- L’I/O si effettua dal dispositivo al buffer locale del controller.
- Il device controller informa la CPU che ha completato la sua operazione generando una interruzione (interrupt).
#### Gestione degli interrupt
I sistemi sono guidati dagli [[3. Linguaggio assembly#Interruzioni|interrupt]], che trasferisce il controllo alla routine di servizio ad esso associata mediante il **vettore degli interrupt**, il quale contiene tutte le routine di servizio.
Il sistema operativo conserva lo stato della CPU a seguito di un interrupt memorizzando i valori dei registri e del program counter.9
>[!important] Un trap è un interrupt software generato da un errore o da una particolare richiesta dell’utente.

![[Pasted image 20260308124134.png|center|600]]
#### Gestione dell'I/O
- **Sincrono**: Dopo che è stata avviata un’operazione di I/O, il controllo ritorna al programma utente soltanto dopo il completamento dell’operazione di I/O.
- **Asincrono**: Dopo che è stata avviata un’operazione di I/O, il controllo ritorna al programma utente senza attendere il completamento dell’operazione di I/O.
  -**System call**, richiesta al sistema operativo per attendere il completamente dell'operazione di I/O
  -**Tabella di stato dei dispositivi**, contiene elementi per ogni singolo dispositivo di I/O che indica il suo tipo, indirizzo e stato. Il sistema operativo accede alla tabella di stato dei dispositivi per determinare lo stato del dispositivo e modificare la sua descrizione.
![[Pasted image 20260308124520.png|center|400]]
#### DMA
Il [[3. Linguaggio assembly#DMA - direct memory access|DMA]] è usato per dispositivi di I/O capaci di trasmettere dati a velocità vicine a quelle della memoria.
Il device controller trasferisce blocchi di dati dal buffer direttamente alla memoria centrale senza l’intervento della CPU.  Il DMA Controller si occupa di gestire gli interrupt intermedi.
Viene generato un solo interrupt per blocco, invece che un interrupt per ogni byte trasferito.
![[Pasted image 20260308124638.png|center|300]]
#### Struttura della memoria
Esistono 2 tipi di memoria
- Memoria centrale
- Memoria secondaria
a loro volta classificate in base a velocità di accesso, costo e volatilità.
![[Pasted image 20260308124909.png|center|300]]
>[!important] caching
>copia dei dati in memorie più veloci; la memoria centrale può essere vista come una cache per la memoria secondaria.
>La cache è controllata prima per verificare se contiene l’informazione di interesse:
>- caso positivo, l’informazione è acceduta direttamente dalla cache (maggiore velocità d’accesso).
>- Altrimenti, i dati sono copiati nella cache e acceduti da lì.

#### Protezione hardware
Il sistema operativo deve impedire che il cattivo funzionamento di un programma influenzi la corretta esecuzione del sistema operativo e di altri programmi.
- **Funzionamento in dual-mode**
  Fornisce un supporto hardware per differenziare tra due modalità principali di esecuzione delle operazioni tramite l'uso di un **bit di modo**:
	1.  Modo Utente – esecuzione svolta per conto dell’utente.
	2.  Modo Monitor (o kernel o system) – esecuzione svolta per conto del sistema operativo.
	Bisogna assicurarsi quindi che un programma utente non possa ottenere il controllo del computer in modo monitor.
- **System call**
  ![[Pasted image 20260308125639.png|center|300]]
- **Protezione della memoria**
  Occorre proteggere aree riservate della memoria come per esempio il vettore delle interrupt.
  Per far questo abbiamo un **registro di base** che contiene l'indirizzo iniziale della memoria che un programma può accedere e un **registro limite** che contiene la dimensione dell'intervallo.
  La memoria al di fuori dell'intervallo è quindi protetta.
  ![[Pasted image 20260308125909.png|center|600]]
- **Protezione della CPU**
  Un processo utente non può usare la CPU a tempo indefinito. Esiste quindi un timer decrementato ad ogni tick del clock, quando arriva a 0 viene generato un interrupt.
  Il caricamento del timer è una istruzione privilegiata per motivi di sicurezza.


>[!bug] Meltdown
>Determinate CPU per aumentare le prestazioni effettuavano controlli a posteriori (in parallelo) e solo successivamente se le autorizzazioni non erano concesse annullavano le operazioni, lasciando però la cache invariata. Questo ha permesso l'accesso a determinate locazioni di memoria a posteriori tramite la loro copia nella RAM.

>[!bug] Spectre
>Sfrutta la predizione delle diramazioni (Il calcolo delle operazioni di if/else venivano effettuate entrambe, ma solo quelle chiamata veniva mantenuta e venivano effettuati i controlli su di essa). Ancora una volta i dati contenuti nella cache, derivati dalla diramazione non attiva, non venivano eliminati e potevano quindi essere visualizzati come nel meltdown. Questo attacco richiede però che il processore abbia una unità di predizione della diramazione (molto comune)

### Interprete dei comandi
Il sistema operativo riceve continuamente comandi tramite istruzioni di controllo che riguardano:
- creazione e gestione dei processi
- gestione dell'I/O
- gestione della memoria secondaria
- gestione della memoria centrale
- accesso al file system
- protezione
- networking
È necessario per il sistema interpretare i comandi ricevuti.
Oggi nella maggior parte dei casi si ha accesso all'interprete dei comandi tramite una interfaccia grafica.
## Servizi di un sistema operativo
- **Esecuzione di programmi** – capacità di caricare in memoria un programma ed eseguirlo.
- **Operazioni di I/O** –  poiché i programmi non possono eseguire direttamente  le operazioni di I/O, il sistema operativo deve fornire meccanismi per le operazioni di I/O.
- **Gestione del File system** – fornire ai programmi modalità per leggere, scrivere, creare e cancellare file.
- **Comunicazioni** – scambio di informazioni tra processi in esecuzione sullo stesso computer o su computer connessi in rete. 
- **Scoperta di errori** – rilevamento di errori nella CPU, nella memoria, nell’I/O o nei programmi utente.

Inoltre esistono funzioni del sistema operativo che non servono per l'esecuzione dei programmi utente, ma per assicurare l'esecuzione efficiente delle operazioni del sistema:
- **Allocazione di risorse** - allocazione di risorse per i programmi eseguiti contemporaneamente
- **Accounting** - memorizzazione dell'uso delle risorse da parte dei vari utenti per il calcolo di costi e per la generazione di statistiche
- **Protezione** - controllo degli accessi a tutte le risorse del sistema

## System call
>[!important] Le system call forniscono l'interfaccia tra un programma utente e il sistema operativo

(In windows si chiamano API, ma non lo sono propriamente)

Il codice delle chiamate non risiede nel programma: una volta richiamata una funzione il controllo passa dal programma al sistema operativo che eseguirà la system call; Questo avviene tramite il cambio di locazione di memoria ([[2. Sistemi di elaboratori#Calcolatore di riferimento|program counter]]?).
Inoltre cambia il tipo di [[#Protezione hardware|protezione]] da modo utente a modo monitor.
Il passaggio di parametro può avvenire tramite:
1. Passaggio di parametri in registri (consentono solo poco spazio)
2. Memorizzazione dei parametri in una tabella in memoria centrale e passaggio dell'indirizzo della tabella, come un parametro, in un registro
3. push dei parametri in uno stack e pop dallo stesso dal sistema operativo (poco comune a causa di problemi di sicurezza, stiamo accedendo allo stack del kernel)
(non esistono più system call per decidere quale utilizzare, è il progettista del kernel a decidere quale implementare e il programmatore si deve adattare)
![[Pasted image 20260308190249.png|center|400]]

Vediamo per esempio la funzione `ReadFile()` delle API di Win32
```
BOOL ReadFile  (HANDLE       file,
				LPVOID       buffer,
				DWORD        bytesToRead,
				LPDWORD      bytesRead,
				LPOVERLAPPED ovl);
```

Le system call permettono di gestire le funzionalità del sistema operativo.
Generalmente vengono utilizzate tramite librerie di linguaggi a più alto livello.
>[!quote] Esempio
>la `prinf()` di [[C]] chiama direttamente la system `call write()`

>[!important] I programmi di sistema forniscono un ambiente conveniente per lo sviluppo e l'esecuzione di applicazioni
>- gestione dei file
>- informazioni di stato
>- supporto ai linguaggi di programmazione
>- caricamento ed esecuzione dei programmi
>- comunicazioni

## Creazione di un sistema operativo
>[!important] Obbiettivi di progetto
>Gli obbiettivi di chi crea il sistema e chi lo deve utilizzare non sempre convergono
>- **Per l'utente**
  >il sistema operativo dovrebbe essere conveniente da usare, facile da apprendere, affidabile, sicuro e veloce
>- **Per il creatore del sistema**
  >dovrebbe essere facile da progettare e manutenere, flessibile, affidabile, libero da errori ed efficiente

>[!important] Meccanismi e politiche
>I **meccanismi** determinano come fare qualcosa, le **politiche** determinano cosa fare.
>La separazione tra meccanismi e politiche è un principio molto importante perché permette la massima flessibilità se le politiche verranno cambiate.
### Implementazione del sistema
I sistemi operativi sono stati sviluppati tradizionalmente in assembler per ragioni di efficienza e di accesso alle risorse hardware.
Oggi vengono implementati tramite linguaggi di alto livello. Il codice scritto in questi linguaggi è:
- più veloce da sviluppare.
- più compatto.
- è facile da comprendere e da correggere.
Un sistema operativo è facile da portare su altro hardware se scritto in un linguaggio di alto livello.
### Debugging dei sistemi operativi
L'attività di trovare e risolvere errori può avvenire tramite la creazione di log contenenti informazioni sull'errore.
Inoltre:
- il fallimento di una applicazione può generare file di **core dump** che catturano la memoria di accesso.
- il fallimento del sistema operativo può generare file di **crash dump** contenenti la memoria del kernel

### Modelli di comunicazione
Lo scambio di informazioni tra processi può avvenire tramite scambio messaggi o memoria condivisa
![[Pasted image 20260309194105.png|center|400]]
- **scambio di messaggi**, tramite la memoria del kernel, questa scrittura avviene tramite il sistema operativo, il processo non ha accesso diretto alla memoria
- **memoria condivisa**, tramite uno spazio di memoria condivisa, entrambi i processi hanno diritto di leggere/scrivere in quelle celle di memoria
Non tutti i sistemi implementano entrambi i metodi.
Inizialmente è stato creato questo meccanismo per i processi propri del sistema operativo, oggi possono essere implementati anche per programmi utenti.
### Approccio a livelli
![[Pasted image 20260309200147.png|center|300]]

Ogni livello utilizza le funzioni del livello sottostante.

## Macchine virtuali
>[!important] Macchine virtuali
>Una macchina virtuale è un ambiente software che emula un computer fisico, consentendo l’esecuzione di più sistemi operativi e applicazioni su un singolo hardware.

Una macchina virtuale porta l'approccio a livelli alla sua massima espressione astratta.
Una macchina virtuale fornisce una identica interfaccia a diverse architetture hardware e più macchine virtuali possono essere installate su uno stesso computer.
Il concetto di macchina virtuale fornisce una protezione completa delle risorse del sistema poiché ogni macchina virtuale è isolata dalle altre. L’isolamento tuttavia non permette condivisione di risorse.
Una macchina virtuale è un ambiente ideale per la ricerca e lo sviluppo che vengono svolti sulla macchina senza perturbare il normale funzionamento del sistema operativo.
Il concetto di macchina virtuale è complesso da implementare a causa della necessità di fornire un duplicato esatto della macchina sottostante.
### Macchine virtuali con so su stesso livello
![[Pasted image 20260309202515.png|center|350]]
Storicamente le macchine virtuali nascono per essere utilizzate su [[#Sistema mainframe|sistemi mainframe]], computer con molti utenti ed esigenze diverse. Si dava all'utente la possibilità di scegliere di utilizzare un sistema operativo piuttosto che un altro sullo stesso elaboratore.
### Macchine virtuali installate sul sistema operativo
![[Pasted image 20260309203058.png|center|200]]
Oggi in realtà è più comune installare la macchina virtuale su un sistema operativo di base, permettendo l'utilizzo di entrambi i sistemi operativi. Ovviamente il sistema operativo su macchina virtuale avrà delle prestazioni minori, dati dal passaggio di dati che deve attraversare la macchina virtuale, il sistema operativo di base per poi arrivare all'hardware.
>[!important] Container
>Una importante evoluzione delle macchine virtuali sono i **container**, delle macchine virtuali che comprendono il codice di un’applicazione e tutte le sue dipendenze (librerie, file di configurazione, runtime). Non comprendono però il sistema operativo in sé, hanno bisogno quindi di essere installati su un so.
### Macchine virtuali e linguaggi di programmazione
[[Java]] è l’esempio perfetto di utilizzo di macchina virtuale nell’uso dei linguaggi di programmazione.
I programmi Java dopo la compilazione sono architecture-neutral e vengono eseguiti dalla Java Virtual Machine (JVM).
Solo una piccola parte della JVM dialoga effettivamente con il sistema operativo, quindi basta implementare quella parte caratterizzata per l'hardware su cui deve girare per rendere utilizzabile tutto java su un elaboratore.
Tramite questo approccio è stata persa dell'efficienza, garantendo però una maggiore portabilità, che ha reso questo linguaggio utilizzabile e molto diffuso per lo sviluppo web.
Se un sito deve caricare del codice java esso gira sulla virtual machine installata nel browser.
