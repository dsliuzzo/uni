# I/O hardware
Data la vastità di sistemi di input-output con cui un sistema operativo si deve interfacciare esistono molti sistemi di gestione raccolti nel [[#Sottosistema I/O del kernel]].

Alcuni elementi sono comuni a tutti i dispositivi:
>[!multi-column]
>
>>[!important] Porta
>>Punto di connessione fisico tra dispositivo e sistema
>
>>[!important] Bus
>>canale di comunicazione condiviso tra più dispositivi. Può essere organizzato a **daisy chain** (dispositivi in serie) o ad **accesso diretto condiviso** (tutti i dispositivi accedono allo stesso bus).
>
>>[!important] Controller
>>chip/circuito (adattatore host) che gestisce il funzionamento del dispositivo e fa da intermediario con la CPU.

Ogni dispositivo è caratterizzato da un **indirizzo** che può essere utilizzato in vario modo:
- **Istruzioni di I/O e registri dedicati**: la CPU usa istruzioni speciali per comunicare con registri del controller (status, control, data-in, data-out).
- **I/O mappato in memoria**: i registri del controller sono mappati nello spazio di indirizzamento della memoria, quindi la CPU li legge/scrive come se fossero normali locazioni di RAM.

## Interrogazione ciclica - polling
L'interazione tra un controller e la CPU avviene tramite il modello [[1. Gestione dei processi#Produttore consumatore|produttore-consumatore]].
Per determinare lo stato del dispositivo si utilizzano 3 bit principali del registro di stato del dispositivo:
- **command-ready**
  periferica pronta a ricevere un nuovo comando
- **busy**
  la periferica sta lavorando e non può ricevere interazioni
- **error**
  segnala se l'operazione richiesta è fallita

La CPU esegue una interrogazione ciclica (busy-wait) finché il bit busy non è impostato a 0. Questo processo occupa inutilmente la CPU.

Per evitare l'attesa inutile vengono quindi utilizzate delle **Interrupt**.
>[!multi-column]
>>[!blank]
>>Se il bit busy è impostato a 0 viene lanciata una `Interrupt` e la CPU esegue una routine di gestione, che si interromperà solo quando il controller terminerà le sue operazioni.
>>
>>Ogni processore ha un suo **vettore** delle interruzioni, basato su priorità, utile per invocare la routine di gestione corretta.
>>Alcuni interrupt sono **mascherabili**: non vengono eseguiti sempre, ma solo in determinate condizioni.
>

## Direct memory access (DMA)
[[3. Linguaggio assembly#DMA - direct memory access]]
Il DMA è un dispositivo hardware (processore) che ha lo scopo di effettuare il controllo nel trasferimento dei dati.
Tramite il DMA la CPU può evitare di occuparsi dai compiti di controllo dei dispositivi di IO.
L'interazione tra il DMA e il controller realizzare il trasferimento.
### Funzionamento del DMA
Esempio di funzionamento di una operazione di IO - spostamento di dati da un disco alla memoria centrale
>[!multi-column]
>>[!blank]
>>1. La **CPU** ordina al **device driver** di trasferire dati dal disco alla memoria nella locazione X
>>2. Il **device driver** ordina al **disk controller** di trasferire una porzione di byte alla memoria
>>3. Il **disk controller** inizializza il trasferimento tramite **DMA**
>>4. Il **disk controller** trasferisce ciascun byte al **DMA**
>>5. Il **DMA** trasferisce i dati direttamente alla locazione X
>>6. Una volta trasferiti tutti i byte alla locazione X, il **DMA** lancia una interrupt alla **CPU** per segnalare il termine della operazione di IO
>
>>[!blank]

# Interfaccia di I/O per le applicazioni
Per far si che tutte le applicazioni possano gestire tutti i diversi comportamenti dei dispositivi in modo omogeneo il sistema operativo fornisce delle [[0. Intro sistemi operativi#System call|system call]] di IO, che offrono una interfaccia uniforme.

>[!important] I driver
>I driver dei dispositivi sono la parte del sistema operativo che permette di usare l'I/O in maniera uniforme (nascondendo le differenze).
>I device driver eseguono le system call comuni adattandole al dispositivo a cui sono associate.

Otteniamo quindi una struttura a più livelli che si sviluppa tra il kernel e i dispositivi hardware

## Tipi di dispositivi di I/O
I dispositivi di I/O vengono convenzionalmente suddivisi in base al loro funzionamento e quindi alla loro gestione
### Dispositivi a blocchi
*es.* [[4. Gestione della memoria secondaria|dischi]]
**operazioni**: `read`, `write`, `seek`
In questi dispositivi l'I/O è di basso livello.
Il file system necessita dell'utilizzo di dispositivi a blocchi.
Permettono di trattare un blocco del disco come se fosse un array nella RAM (**accesso ai file mappato in memoria**).
### Dispositivi a caratteri
*es.* tastiera, mouse, porte seriali
**operazioni**: `get`, `put`
Si possono costruire operazioni più strutturate su buffer di caratteri (il dispositivo legge un carattere alla volta, ma il driver può raggrupparli fornendo per esempio una intera riga).
### Dispositivi di rete
La rete non segue divisione in blocchi o caratteri a causa dell’imprevedibilità del routing dei pacchetti:
- schede di rete non si prestano al posizionamento (come blocchi)
- trasmissione in pacchetti non è un puro flusso continuo locale (come caratteri)

SO come Unix e Windows NT/9x/2000 includono una **socket interface**.
Socket è un'astrazione che risolve questo problema:
- Separa i protocolli di rete dalle operazioni di rete: ignora il tipo di protocollo (TCP, UDP o IP)
- Include `select` per controllare più porte: `select()` è una system call vitale nei server.     
Permette a 1 singolo processo di monitorare contemporaneamente tante connessioni di rete in attesa di dati, senza bloccarsi su una sola
I dispositivi di rete vengono utilizzati per diversi scopi: pipes, code FIFO, stream sono dei meccanismi di comunicazione tra processi che il SO sfrutta per far funzionare 2 programmi su macchine diverse collegate sulla rete.
### Clock e timer
Forniscono l'ora corrente, il tempo trascorso da un dato evento, la regolazione del timer per indicare quando avviene una determinata operazione.
I timer programmabili permettono di avviare operazioni o interrupt in dati intervalli temporali.
In Unix la system call `ioctrl` nelle sue diverse forme permette di gestire operazioni con i timer.
## Sottosistema I/O del kernel
Il kernel del sistema operativo offre un insieme di servizi per la gestione delle operazioni di I/O.
>[!multi-column]
>
>>[!important] Scheduling
>>Permette la gestione delle code delle richieste di I/O
>
>>[!important] Buffering
>>Memorizza dati in memoria centrale durante il loro trasferimento tra i dispositivi.
>>Risulta molto utile per "sincronizzare" il funzionamento di dispositivi con velocità o dimensioni di memoria diverse.
>

La chiamata ad una system call per la richiesta di una operazione di I/O da parte di un processo genera quindi una serie di eventi più complessi:

>[!multi-column]
>
>>[!blank]
>>Per prima cosa il sottosistema di I/O verifica se la richiesta può essere gestita immediatamente, se questo non è possibile manda la richiesta al device driver e blocca il processo.
>>Il device driver quindi inizia le operazioni di I/O come descritto in precedenza. Quando si concludono le operazioni di I/O viene lanciata una interrupt. Il device driver comunica al kernel quale operazione di I/O è stata completata, a quel punto il kernel trasferisce i dati al processo e ritorna la system call.
>>Il processo può quindi continuare la sua esecuzione.
>
>>[!blank]

# Prestazioni
Le operazioni di I/O incidono fortemente sulle prestazioni di un processo a causa:
- l'esecuzione dei driver
- i numerosi context switch
- la gestione dello scheduling cazzo
- grandi quantità di dati da trasferire

Di conseguenza per migliorare le prestazioni di una applicazione è necessario:
- ridurre il numero di context switch
- ridurre la copia di dati
- aumentare le quantità di dati trasferiti per singolo interrupt, riducendo il numero totale di interrupt
- usare il DMA
- bilanciare il carico di CPU, memoria, bus e sottosistema di I/O
