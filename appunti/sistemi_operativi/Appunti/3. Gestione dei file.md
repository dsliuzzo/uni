#so 
Un file è un insieme logico di informazioni.
Le **directory** sono file particolari.
La componente della gestione dei file da parte del sistema operativo è responsabile di:
- Creazione e cancellazione dei file
- Creazione e cancellazione delle directory
- supporto di primitive per manipolare file e directory
- mappare file in memoria secondaria
- backup dei file su memorie non volatili
La struttura dati dei file vista dai programmi si appoggia alla gestione dei file del sistema operativo.
# File
>[!Important] File
>Un file è uno spazio di memoria logicamente contiguo contenuto all’interno del disco a cui è assegnato un nome che rappresenta un documento, una scheda o un’unità di informazione.

Un file, a livello fisico sul disco, è solo una sequenza di bit. Tuttavia, per essere utile, deve avere un'organizzazione logica che permetta di interpretare bit. La struttura di un file definisce proprio questa organizzazione. 
Possiamo classificare le strutture in tre grandi categorie:
1. **Nessuna struttura** (Flusso di word o byte): È l'approccio più semplice e flessibile. Il SO non impone alcuna formattazione: file visto solo come una sequenza continua di byte. 
   Sarà il programma applicativo a sapere come leggere e interpretare quei byte. 
2. **Struttura semplice a record**: File visto come una sequenza di record (righe o schede). I record possono avere una lunghezza fissata o una lunghezza variabile.
3. **Struttura complessa**: File ha organizzazione interna articolata (tipico di documenti formattati o i dati rilocabili/file eseguibili).
Nei SO moderni che adottano l'approccio "nessuna struttura", è possibile simulare le strutture a record inserendo dei speciali caratteri di controllo.
Questa struttura può essere imposta dal
- **so** che deve supportare nativamente ogni formato
- **programma applicativo** il so fornisce solamente i byte grezzi al programma
## Attributi
Per il SO, un file è composto da due entità distinte:
- Dati: contenuto vero e proprio del file 
- Metadati: attributi che descrivono il file
Gli attributi includono: 
- Nome simbolico (quello leggibile dall'utente, come appunti.pdf)
- Tipo: serve al SO o ai programmi applicativi per capire come interpretare il contenuto. 
  Infatti, un file eseguibile (.exe) conterrà istruzioni in linguaggio macchina, mentre un file sorgente (.c) conterrà testo leggibile da un compilatore.
- Posizione fisica sul disco
- Dimensione
- Informazioni temporali (data di creazione, ultima modifica)
- Bit di protezione (chi può leggere, scrivere o eseguire il file) 
Tuttavia, questi attributi non sono salvati dentro il file, ma sono custoditi all'interno di una directory.
## Operazioni sui file
Per manipolare questi file, il SO utilizza delle system call standard: Creazione, Scrittura, **Lettura**, **Copia** e **cancellazione**. Esistono anche:
- **Riposizionamento**:  permette di spostare il "puntatore" interno del file senza dover leggere tutto il contenuto intermedio
- **Troncamento**: Svuota completamente il contenuto di un file portando la sua dimensione a zero, ma senza cancellarne gli attributi.
Tuttavia, prima di poter compiere qualsiasi operazione continuativa (come leggere o scrivere molte volte), il file deve essere aperto:
- `Open(Fi)`: cerca il file nella directory su disco e copia le sue informazioni di controllo in memoria centrale, evitando al SO di accedere al disco fisso per ogni singola lettura (sarebbe troppo lento)
- `Close(Fi)`: aggiorna il disco e libera la memoria centrale al termine delle operazioni (non è necessario aggiornare il disco se non sono stati effettuati cambiamenti)
## Metodi di accesso
Una volta che le informazioni sono state memorizzate in un file, devono poter essere lette e scritte, attraverso dei metodi di accesso.
Esistono due modalità principali:
>[!multi-column]
>
>>[!important] Accesso sequenziale
>>Informazioni vengono elaborate in ordine, un record dopo l'altro
>>Le operazioni primarie sono:
>> - read next: lettura del successivo
>> - write next: scrittura nello spazio successivo
>> - reset: torno all’inizio del file
>>
>>Non si può saltare ad un record senza aver prima letto precedenti.
>>![[3. Gestione dei file-1776787619693.webp|center|300]]
>
>>[!important] Accesso diretto
>>File visti come sequenza numerata di blocchi o record logici di lunghezza fissa. Un programma può leggere o scrivere i blocchi in qualsiasi ordine in modo rapido:
>>Le operazioni primarie sono (dove n è il numero del blocco):
>> - read n: lettura del successivo
>> - write n: scrittura nello spazio successivo
>> - position to n: posso procedere con read n e write n

>[!question] Osservazione
>I sistemi moderni, basati su dischi ad accesso diretto, possono facilmente simulare l'accesso sequenziale mantenendo internamente una variabile di stato(cp - current position) che viene incrementata automaticamente dopo ogni lettura/scrittura.
>![[3. Gestione dei file-1776787682363.webp|center|300]]
>File Indice: utilizzati per velocizzare la ricerca in file molto vasti. L'indice contiene dei puntatori ai vari blocchi del file principale: per trovare un dato, si cerca prima la chiave nell'indice e poi si usa il puntatore per accedere direttamente al blocco del file.
>![[3. Gestione dei file-1776787871731.webp|center|300]]
# Directory
>[!important] Directory
>è definita come una collezione di nodi, dove ogni nodo contiene le informazioni (metadati) per accedere a uno specifico file.

è una speciale struttura dati utilizzata dal S.O. per mantenere un registro dei file: viene spesso chiamata "cartella" nei S.O. con interfaccia grafica.
La directory sono memorizzate come dei file speciali e viene caricata in RAM solo quando è necessario.
## Organizzazione del file system
I dischi rigidi non vengono quasi mai utilizzati come un unico blocco, ma vengono suddivisi in **partizioni** (o volumi), che il S.O. tratta come se fossero dischi separati e indipendenti.
![[3. Gestione dei file-1776788180768.webp|center|400]]
## attributi della directory
La directory mantiene una tabella per i file contenuti della partizione e per ognuno registra:
- **Nome e Tipo**: Come l'utente lo chiama e in che formato è scritto.
- **Indirizzo**: I puntatori fisici o logici ai blocchi del disco dove risiedono i dati.
- **Lunghezza corrente e massima**: La dimensione attuale e, in alcuni sistemi operativi più vecchi o specializzati, il limite di spazio allocabile per quel file.
- **Date e Ora**: Ultimo accesso (lettura), aggiornamento (scrittura) e la data di creazione.
- **ID del proprietario e Protezione**: Chi crea il file e quali permessi (Lettura/Scrittura/Esecuzione) sono concessi agli altri utenti del sistema.
## operazioni sulle directory
Le directory sono strutture dati dinamiche su cui il SO deve fornire un set di operazioni fondamentali: 
- **Ricerca** `find`: scansionare la directory per trovare l'indirizzo di un file dato il suo nome.
- **Creazione/Cancellazione di un file** `rm`: quando si crea un file si aggiunge un nodo alla directory, mentre se lo si cancella, si rimuove quel nodo
- **Elencazione (List)** `ls`: leggere contenuto della directory per mostrare i file e le loro proprietà 
- **Rinominazione** `mv`: modificare solo la stringa del nome nel nodo della directory
- **Creazione/Eliminazione di una directory** `mkdir`: le directory possono contenere altre directory (alberi)
- **Attraversamento (Traverse)** `-r`: Un'operazione usata per visitare sistematicamente ogni singola directory e sottodirectory dell'intero file system.
## Organizzazione logica di una directory
Sono tre gli obiettivi principali che un buon file system deve garantire:
1. **Efficienza**: Localizzare un file deve essere rapido
2. **Identificazione flessibile** (Naming): Il sistema deve essere conveniente per gli utenti:
	- Due utenti possono usare nomi uguali per file diversi (senza conflitto)
	- Lo stesso file fisico potrebbe essere accessibile tramite nomi diversi.
3. **Raggruppamento logico**: Utenti devono poter categorizzare i file in base alle proprietà (es. tutti i programmi Java, tutti i videogiochi, ecc.)
Per soddisfare queste esigenze, le directory possono essere organizzate in diverse strutture dati:
### Directory a livello singolo
Tutti i file di tutti gli utenti risiedono nell'unica directory principale (root). Questo approccio fallisce immediatamente:
- Identificazione: nomi dei file sono globalmente univoci nel sistema (causa collisioni tra utenti)
- Raggruppamento: tutto è ammassato e non ordinato secondo criteri in un’unica lista
![[3. Gestione dei file-1776841854156.webp|center|600]]
### Directory a due livelli
Nato per risolvere il problema delle collisioni, introducendo una Master File Directory (MFD) che punta a singole User File Directory (UFD). Ogni utente ha la sua directory personale, per cui due utenti possono avere un file chiamato `test.c` senza conflitti.
![[3. Gestione dei file-1776841997374.webp|center|600]]
Tuttavia, il raggruppamento è ancora impossibile (l'utente non può creare sottocartelle).
>[!important] Path name
>Per evitare i conflitti nasce il concetto primitivo di Path name per identificare univocamente combinando utente e specifico file (es. `Utente1/test.c`).
### Directory con struttura ad albero
La struttura è un grafo ad albero rovesciato, che parte da una radice e si ramifica, realizzando l’idea che una directory può contenere file e  sottodirectory.
I nodi intermedi sono le directory, mentre le foglie sono i file veri e propri.
Questa struttura risolve il problema dei nomi, garantisce l'efficienza organizzativa e permette un raggruppamento gerarchico infinito.
![[3. Gestione dei file-1776842110642.webp|center|600]]
Per navigare l'albero viene ampliato il concetto di Path name
>[!important] Working directory
>Nodo in cui si trova il processo attivo

>[!multi-column]
>
>>[!important] Path name assoluto
>>Il percorso completo, passo dopo passo, dalla radice fino al file bersaglio (rende un indirizzo inequivocabile) → `cd/spell/dist`
>
>>[!important] Path name relativo
>>Dalla Directory Corrente, descrivo il percorso ignorando la radice.

Viene introdotto `mkdir` per la creazione della directory e `rm` per eliminarla. In molti sistemi operativi può essere eliminata solo se vuota, altrimenti richiede un comando di cancellazione esplicita, per evitare cancellazioni accidentali.
### Directory a grafo aciclico
[[Strutture dati#Grafi]]
La struttura ad albero è ottima per l'organizzazione, ma non permette condivisione diretta. 
In un albero puro, un file ha un solo padre. Se volessi lo stesso file in più cartelle senza farne una copia, l'albero puro non lo permetterebbe. 
Per questo si passa a una struttura a grafo aciclico, dove i rami possono intrecciarsi e non ci sono percorsi chiusi perché non posso risalire.
Ho la possibilità di mantenere lo stesso file su più cartelle diverse senza doverlo copiare e mantenere sul disco due volte.
![[3. Gestione dei file-1776842668097.webp|center|600]]


>[!bug] Aliasing
>percorsi diversi portano allo stesso file → stesso file può avere nomi completamente diversi a seconda di chi lo sta guardando (complica ricerche)

>[!bug] Tangling pointers
L'eliminazione di un file diventa più difficoltosa: se un file è condiviso da più directory la sua eliminazione lascerebbe un riferimento appeso nella directory a cui era condiviso il file.
Per ovviare a questo problema possiamo utilizzare dei **puntatori all'indietro**, che ci permettono di eliminare tutti i riferimenti al file; oppure ogni nodo contiene un **reference counting** che conta il numero di link a quel file.
### Directory a grafo generale
Per poter creare collegamenti a qualsiasi cosa, comprese le directory superiori, si creeranno cicli all’interno del grafo. 
Infatti, nella ricerca di un file, il sistema inizia a esplorare l’albero, rimanendo incastrato per in questo loop circolare.
>[!bug] Creazione di cicli
>[[Strutture dati#Verifica aciclicità grafi orientati]]
>Per garantire l'assenza di cicli:
>- Creazione di link condivisi solo per i file Invece, sono proibiti i link alle sotto-directory: senza link è impossibile creare loop.
>- **Garbage Collection**: Si permette di fare tutto, ma periodicamente il sistema si ferma e analizza il disco per scovare cicli inutilizzati o incastrati e  distruggerli (soluzione lentissima e costosa).
>- **Algoritmi di rilevazione**: Ogni volta che l'utente tenta di creare un link, SO calcola la possibilità di generare un ciclo e, in caso affermativo, blocca l'operazione (troppi controlli → lento). Con questa implementazione è esattamente un grafo aciclico

## Mounting del file system
Se il SO ha necessità di utilizzare file su dischi diversi, bisogna effettuare un’operazione di **mounting**: connessione logica dei nuovi file nell’albero principale. Solo dopo aver effettuato questa operazioni i nuovi file possono essere acceduti. 
Al contrario, quando scollego un disco devo fare un’operazione di **unmounting**: si scollega dal resto del grafo e quel pezzo del file system non esisterà più. Questa operazione deve essere fatto dall’utente, in quanto non è automatico: nel caso in cui non venisse effettuato il SO genererebbe un’interrupt.
![[3. Gestione dei file-1776844294758.webp|center|400]]
>[!important] Mount point
>è il punto di montaggio ovvero la directory esistente nel file system principale scelto dal SO o dall’utente.
>Il SO lega la radice del disco sopra il nodo /users e se l'utente naviga nel percorso, starà leggendo i dati dal secondo disco, ma senza accorgersene, perché tutto sembrerà far parte di un albero continuo.
## Condivisione di file
Nei sistemi distribuiti serve condividere file tra più utenti o più macchine attraverso una rete.
Per farlo in sicurezza, è obbligatorio implementare schemi di protezione molto rigidi.
>[!quote] Esempio: NFS (Network File System)
>è un modello e una realizzazione pratica di file system distribuito, che permette a un computer di  "montare" una directory che risiede fisicamente sull'hard disk di un server dall'altra parte del mondo. 
>L'utente apre i file come se fossero sul suo PC locale: è la rete che trasmettere i dati in modo invisibile.
### Protezione dei file
[[6. Sistema di protezione e sicurezza]]
Poiché i file possono essere condivisi, il proprietario del file deve poter decidere:
1. **Tipo di accesso**: Lettura, Scrittura, Esecuzione per i programmi, Aggiunta di dati alla fine senza cancellare il resto, Cancellazione totale, Lista per vedere il contenuto delle directory
2. **Chi** può eseguire queste operazioni sui file condivisi
>[!question] Il proprietario può togliersi e ridarsi i diritti liberamente

  Due metodi di protezione poco utilizzati sono:
- **Password**: ogni singolo file è protetto da una password (obsoleto e impraticabile con tanti file)
- **Liste di accesso**: lista dettagliata allegata al file che indica per ciascun utente quali sono i suoi diritti (preciso ma richiede memoria per memorizzare liste)
#### Gruppi
Per risolvere il problema della pesantezza delle ACL, sistemi come Unix usano un sistema matematico basato su 3 classi e 3 permessi. 
I permessi base sono: Read (r), Write (w), Execute (x). 
Gli utenti del sistema sono raggruppati in tre classi concentriche:
1. Proprietario (Owner): Chi ha creato il file.
2. Gruppo (Group): Un insieme di utenti con mansioni simili creato dal proprietario del file.
3. Pubblico (Others): Tutti gli altri utenti che hanno accesso ma non rientrano nei primi due.
Il sistema usa matematica binaria per comprimere le informazioni, assegnando dei pesi ai permessi:
- Read (R) vale 4
- Write (W) vale 2,
- Execute (X) vale 1
Sommando questi pesi, si ottiene un numero che definisce una combinazione di permessi:

| Tipologia di accesso | $n°$        | R   | W   | X   | Permessi                        |
| -------------------- | ----------- | --- | --- | --- | ------------------------------- |
| accesso proprietario | $7 = 4+2+1$ | 1   | 1   | 1   | permessi totali                 |
| accesso gruppo       | $5 = 4+0+1$ | 1   | 0   | 1   | leggere, eseguire, non scrivere |
| accesso pubblico     | $1=0+0+1$   | 0   | 0   | 1   | solo eseguire                   |
>[!question] Differenza con le liste di accesso
>a differenza delle liste di accesso, ad ogni utente è associato una tipologia di accesso, ma globalmente, l'informazione non è associata ad ogni file.
# Implementazione del file system
>[!important] L'implementazione fisica dei file non corrisponde alla visione logica

Il file system si infrappone (logicamente) tra applicazioni e dispositivi e si sviluppa su più livelli:
![[3. Gestione dei file-1777016624822.webp|center|300]]
- <font color="grey"  size="4"> Applicazioni </font>
- **File system logico**
  presenta il file system come un'unica struttura; implementa i controlli di protezione
- **Modulo di organizzazione dei file**
  controlla l'allocazione dei blocchi fisici e la loro associazione con quelli logici: traduzione da indirizzi logici a fisici
- **File system di base**
  usa i driver per accedere ai dispositivi di I/O
- **Controllo dell'I/O**
  è composto dai driver: gestisce il continuo trasferimento dei dati (non vede il file - sono i blocchi superiori ad associare i dati ad un file)
- <font color="grey"  size="4"> Dispositivi </font>
## File control block
Il **file control block** è una struttura dati che contiene tutti i [[#Attributi|metadati]] dei file (su unix viene chiamato *inode*).

Ogni operazione sui file ha bisogno di accedere al file control block.

-  **Apertura dei file**  
>[!multi-column]
>
>>[!blank]
>>All'apertura di un file accediamo alla struttura della directory, che contiene il riferimento al file control block, se non è presente in memoria centrale viene prima caricata la directory e poi può essere ricercato il FCB.
>
>>[!blank]

>[!attention] Al momento della open i file non vengono caricati in memoria centrale, ma viene solo preparata l'infrastruttura per poter accedere al disco.

Possiamo implementare l'allocazione in vario modo
### Allocazione contigua
Con questa implementazione spesso i blocchi occupano la stessa memoria.

>
>>[!check] Vantaggi
>>- è sufficiente mantenere indirizzo iniziale e lunghezza
>>- accesso diretto ai file
>
>>[!bug] Svantaggi
>>- spreco di memoria
>>- è necessario trovare una locazione vuota sufficientemente grande
>>- spesso non possono crescere di dimensione

Per ricavare l'indirizzo logico partendo dall'indirizzo fisico, si divide (divisione intera) per la dimensione del singolo blocco e sommo lo spiazzamento (posizione relativa).
#### Allocazione contigua modificata
Possiamo risolvere il problema dell'impossibilità di espandere la dimensione del file aggiungendo (in caso di ampliamento) una nuova locazione contigua (**extent**) nello spazio libero del disco.
Per questa implementazione è necessario mantenere il primo indirizzo degli extent e avere un contatore degli extent.
### Allocazione concatenata
Utilizza praticamente una linked list di blocchi su disco, in questo modo fisicamente possono trovarsi in punti diversi.
>[!multi-column]
>
>>[!check] Vantaggi
>>- semplice da implementare (necessita solo del puntatore iniziale)
>>- non abbiamo spreco di memoria
>>- i file possono crescere
>
>>[!bug] Svantaggi
>>- se si spezza la catena di puntatori non posso leggere il file
>>- non supporta l'accesso diretto (per raggiungere un punto specifico devo scorrere l'intero file)

![[3. Gestione dei file-1777019037771.webp|center|400]]
Per il mapping di un blocco di 512 word potrà contenere 511 word di dati, in quanto deve riservare un word al blocco successivo.
#### Allocazione concatenata con file allocation table
MS-DOS utilizzava una variante dell'allocaizone concatenata: il FAT (File allocation table)
>[!important] File allocation table
>Tabella contenente tanti elementi quanti sono i blocchi su disco, ogni elemento contiene l'indice del prossimo blocco.
>![[3. Gestione dei file-1777020874320.webp|center|600]]

Manteniamo la catena dei puntatori in RAM e cerchiamo il percorso in memoria centrale, una volta arrivati al puntatore corretto passiamo al disco, ottimizzando il tempo di accesso, grazie alla ricerca molto più veloce in RAM, piuttosto che direttamente du disco.
Una volta trovato il blocco corretto utilizziamo l'indice della file allocation table viene utilizzato l'indice, moltiplicato per la grandezza della tabella, per trovare la locazione su disco.
Quindi ci deve essere una chiara corrispondenza tra la grandezza della FAT e la grandezza del disco e, almeno a livello logico, deve essere implementata in modo contiguo.

### Allocazione indicizzata
Manteniamo una struttura dati contenente tutti i puntatori ai blocchi dei file: il **blocco indice**
>[!multi-column]
>
>>[!check] Vantaggi
>>- accesso diretto alle informazioni
>>- non c'è frammentazione
>
>>[!bug] Svantaggi
>>- il blocco indice occupa spazio

![[3. Gestione dei file-1777021856975.webp|center|400]]
Il file control block contiene l'indirizzo del blocco indice.
Per il mapping si divide l'indirizzo logico per la dimensione del blocco: il quoziente rappresenta lo spiazzamento nel blocco indice, a quel punto utilizzo il resto per lo spiazzamento nel blocco del file.

>[!bug] Il blocco indice potrebbe non essere sufficiente per indicizzare tutti i blocchi del file
>Possiamo risolvere implementando (al blocco indice):
>- **schema concatenato**
>  l'ultimo elemento del blocco indice punta al successivo
>- **schema multilivello**
>  Abbiamo una suddivisione: il blocco indice di primo livello contiene solo puntatori ad altri blocchi indici (di secondo livello) i quali punteranno effettivamente ai blocchi dati
>- **schema combinato**
> Implementato dagli inode di linux: una parte dei puntatori punta direttamente ai blocchi dati, mentre gli ultimi puntatori puntano ad altri blocchi indice, che possono essere doppi o singoli quindi di maggiore dimensione.

## Gestione dei blocchi liberi
I blocchi liberi possono essere gestiti:
- **Vettore di bit** (bit map) composto da $n$ bit per $n$ blocchi: 1 per i blocchi liberi e 0 per i blocchi occupati.
  Richiede molto spazio, quindi potrebbe non trovarsi tutto in memoria:
	  con una dimensione del blocco $2^n$ e una dimensione del disco $2^m$, la bit map occuperà $\frac{2^m}{2^n}=2^{m-n}$ bit.
  Per trovare il primo elemento libero posso saltare tutte le prima word che non hanno blocchi liberi:
	  `(numero di bit per word) * (numero di word con valore 0) + offset del primo bit a 1`
- **Lista concatenata di blocchi liberi**
  Tutti i blocchi liberi sono concatenati tra di loro.
  Risulta più efficiente a livello di spazio della bit map, ma è meno efficiente nella ricerca.
  Invece di avere un puntatore per singolo blocco potremmo linkare insiemi di blocchi, per aumentare l'efficienza.
# File system distribuiti
Un **file system distribuito** è un file system residente su computer differenti che offrono una vista integrata dei dati memorizzati su diversi dischi in remoto.
Fisicamente sono su macchine differenti, ma logicamente sono visti come un unico disco.
## NFS (network file system)
Permette di fare il mounting di diversi file system: ogni file server fornisce una vista unificata del suo file system locale. Inoltre può essere usato su gruppi eterogenei di sistemi.
Per essere implementato necessita di una mappa di tutti i file (su ogni dispositivo) per capire quali file sono presenti in locale e quali sono sulla rete.
Un accesso locale potrebbe risultare più veloce dell'accesso remoto.
